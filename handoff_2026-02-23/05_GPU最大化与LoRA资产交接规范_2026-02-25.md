# GPU 最大化与 LoRA 资产交接规范（2026-02-25）

本规范用于后续所有 cross-model LoRA 训练与测评，目标是两件事：

1. 在 RTX PRO 6000 96GB 上稳定跑满算力（不牺牲实验可复现性）。
2. 任何 AI/工程师接手时，能在 1 分钟内找到基础模型、LoRA、inv_freq 并完成评测。

## 1. 硬性执行规则（必须遵守）

- 使用新链路，不改老脚本：
  - 训练启动：`scripts/cross_model_finetune_fast_tuned.sh`
  - 训练主体：`scripts/train_cross_model_lora_fast_tuned.py`
- 默认训练档位（已验证可稳定运行）：
  - `max_steps=400`
  - `per_device_train_batch_size=1`
  - `gradient_accumulation_steps=1`
  - `gradient_checkpointing=1`
  - `load_in_4bit=1`
  - `max_seq_len=16384`
- tuned 参数固定：
  - `anchor_factor=4`
  - `slope_raw=20`
  - `center_ratio=0.70`
- 训练完成后必须刷新资产索引（registry）：
  - `python scripts/build_model_registry.py`
- 评测必须显式记录 base + LoRA + custom_inv_freq 三元组，禁止仅凭“方法名”猜路径。

## 2. 目录合同（Artifact Contract）

每个 run 必须满足以下结构：

```text
<run_dir>/
  final_lora/                     # LoRA adapter
  artifacts/
    custom_inv_freq.pt            # 注入频率张量
    summary.json                  # 训练摘要（含 hyperparams + inv_sha256）
  logs/
```

其中 `summary.json` 最低必备字段：

- `base_model_path_resolved`
- `method`, `seed`
- `hyperparams.max_steps`
- `hyperparams.per_device_train_batch_size`
- `hyperparams.gradient_accumulation_steps`
- `hyperparams.gradient_checkpointing`
- `rope.inv_sha256`
- `rope.schedule_params.anchor_factor_effective`
- `rope.schedule_params.slope_raw`
- `rope.schedule_params.center_ratio`

## 3. 一键生成资产索引（给 AI 交接）

在仓库根目录执行：

```bash
python scripts/build_model_registry.py
```

生成目录：`artifacts/model_registry/`

- `base_models.tsv`: 基础模型清单
- `lora_runs.tsv`: 全部 LoRA 运行（含不完整）
- `lora_weights.tsv`: 可直接测评的完整产物（summary + final_lora）
- `lora_partial.tsv`: 不完整运行（排障用）
- `latest_by_method.tsv`: 每个模型/方法最新可用 run
- `README.md`: 快速加载评测命令

## 4. 标准评测加载方式（任何 AI 通用）

从 registry 选定某个 `run_dir` 后，用以下方式评测：

```bash
python scripts/run_eval.py \
  --exp cross_model_eval \
  --model <BASE_MODEL_PATH> \
  --method anchored_sigmoid \
  --ctx 16384 \
  --seed 42 \
  --adapter_override <RUN_DIR>/final_lora \
  --custom_inv_freq_path <RUN_DIR>/artifacts/custom_inv_freq.pt \
  --suite ppl,longbench_full,needle \
  --longbench_score_scale pct
```

必须同时传：

- `--adapter_override`
- `--custom_inv_freq_path`

这样可避免方法映射和目录命名误差。

## 5. GPU 利用策略（预算优先）

- 先稳态再提速：先用默认档位验证 50 steps 无 OOM，再考虑 `PER_DEVICE_BATCH=2`。
- 若 OOM，严格按以下回退顺序：
  1. `PER_DEVICE_BATCH` 降到 1
  2. 保持 `GRAD_ACCUM=1`
  3. 保持 `GRAD_CHECKPOINTING=1`
- 不再使用“batch 小 + 大梯度累积”制造慢速训练。
- `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True` 保持开启，降低显存碎片导致的伪 OOM。

## 6. 验收清单（提交前）

- 训练脚本默认参数与本规范一致。
- 至少 1 个最新 run 在 `lora_weights.tsv` 可见。
- `latest_by_method.tsv` 中能定位当前推荐 run。
- 使用 registry 给出的路径，`run_eval.py` 可直接启动。
- 文档已说明 tuned 参数和 400-step 决策。
